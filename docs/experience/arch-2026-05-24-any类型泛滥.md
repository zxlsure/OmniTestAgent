# ARCH-001: any 类型泛滥

## 现象
全项目约 60+ 处 `any` 类型使用，导致类型系统形同虚设。

## 根因分析
1. **useIpc.ts 全局 any**：`Window.electronAPI` 的所有方法参数和返回值均为 `any`，这是类型污染的根源，传播到所有 Store 和组件
2. **DbAdapter 全部 any**：`private db: any`、`run(sql, params: any[])`、`get()/all()` 无泛型
3. **SkillContext/SkillResult 全部 any**：`llmService?: any`、`output?: any`、`artifacts?: any[]`
4. **IPC handler 参数**：所有 `params: any`、`data: any`
5. **Store 中 ref<any>**：`ref<any>(null)` 代替具体类型

## 改进方案
1. 重写 `useIpc.ts` → `types/electron-api.d.ts`，为每个 IPC 方法定义严格类型
2. DbAdapter 泛型化：`get<T>()`、`all<T>()`
3. SkillContext 强类型化
4. IPC handler 定义请求/响应类型对
5. Store ref 使用具体类型

## 经验教训
1. **类型安全从接口定义开始**：IPC 桥接层的类型定义是整个调用链的类型根基
2. **any 像病毒一样传播**：一处 any 会污染所有调用方
3. **宁可多写类型定义，也不要用 any 图省事**
