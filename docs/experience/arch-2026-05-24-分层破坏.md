# ARCH-002: 分层架构被破坏

## 现象
多处跨层调用违反了 IPC→Service→Repository→Database 的分层原则。

## 具体位置
1. **IPC→Data 跨层**：`storeHandler.ts` 直接导入 `data/secureStore`，绕过 Service 层
2. **Service→Data 跨层**：`KnowledgeService.search()` 直接 `require('../data/database')` 并写原生 SQL
3. **Repository 跨领域**：`ProjectRepo.getStats()` 直接查询其他领域表（test_case, test_flow_activity, knowledge_base）
4. **业务逻辑下沉**：`SkillRepo.toggle()` 中"核心Skill不可全部禁用"规则不应在 Repository 层

## 改进方案
1. 新建 StoreService，storeHandler 调用 Service 层
2. KnowledgeService.search() 的 SQL 移入 KnowledgeRepo
3. ProjectRepo.getStats() 改为 Service 层聚合多个 Repo 结果
4. SkillRepo 的业务规则上移至 SkillRegistry/SkillService

## 经验教训
1. **分层架构的价值在于约束**：每层只与相邻层交互，跨层调用应视为架构bug
2. **业务规则属于 Service 层**：Repository 只做数据存取，不含业务逻辑
3. **跨领域查询在 Service 层聚合**：一个 Repo 不应查询另一个 Repo 的表
