# ARCH-005: 异常处理缺失

## 现象
几乎所有异步操作缺乏异常处理，IPC 通信一旦失败将导致 UI 状态死锁。

## 具体位置
### 主进程
- 7处非空断言无运行时保障（mainWindow!、getById()!、getActive()!、getFocusedWindow()!）
- IPC registerAllIpcHandlers() 中动态 import 未 await
- McpService 中 JSON.parse 无 try/catch
- TestFlowOrchestrator 中 JSON.parse(input_data) 无 try/catch

### 渲染进程
- 几乎所有 Store 的 IPC 调用无 try/catch
- ChatStore.sendMessage 无异常处理，streaming 状态可能永远为 true
- ChatStore 流式监听 removeStreamListeners 可能不被调用（内存泄漏）
- ProjectStore.deleteProject 删除失败仍从列表移除
- SkillsView.toggleSkill 空 catch 吞没异常
- ProjectManagementView.onDelete 无 try/catch

## 改进方案
1. 主进程：所有非空断言改为显式空值检查 + 抛有意义错误
2. 渲染进程：新建 useIpcCall composable 封装统一异常处理
3. ChatStore：sendMessage 添加 try/catch/finally，确保 streaming 和 listener 清理
4. 所有 Store：IPC 调用包装异常处理，失败时保持状态一致性

## 经验教训
1. **异步操作必须 try/catch**：未捕获的 Promise rejection 会导致不可预测的行为
2. **资源清理在 finally 中**：streaming 标志、listener 移除等必须放在 finally 块
3. **空 catch 块是反模式**：吞没异常比抛出异常更危险，用户无反馈且状态不一致
4. **非空断言是运行时炸弹**：TypeScript 的 ! 不提供运行时保障，只是绕过编译检查
